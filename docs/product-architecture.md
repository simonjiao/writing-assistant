# 产品架构

## 工作台布局

```text
┌──────────────────────────────────────────────┐
│ 顶部：Workflow 状态 / Queue 状态 / 保存状态    │
├──────────────┬─────────────────┬─────────────┤
│ 左侧任务卡     │ 中间文章编辑区     │ 右侧知识面板 │
│ Task Card     │ Article Blocks   │ Sources/Tags│
├──────────────┴─────────────────┴─────────────┤
│ 底部：自然语言反馈 / 局部修改 / Patch 预览       │
└──────────────────────────────────────────────┘
```

## 用户交互对象

| 对象 | 说明 |
|---|---|
| Task Card | 写作目标、范围、风格、约束、修改策略 |
| Outline | 文章结构和章节目标 |
| Article Block | 标题、章节、段落、引用和标签绑定 |
| Knowledge Source | RAG 检索结果或本地知识条目 |
| Patch | 针对选中 block 的局部修改方案 |
| Version | 任务卡、大纲、正文、patch 的提交记录 |

## 交互原则

- Chat 表达意图，Workspace 承载结果。
- 默认围绕选中内容操作，不默认全文重写。
- 任务卡、大纲、patch、扩大修改范围需要用户确认。
- 右侧知识面板随全文/章节/段落上下文变化。
- 所有可复用结果写入 ArtifactStore。

## 关键用户流程

```text
1. 用户输入模糊需求。
2. 系统生成任务卡草稿。
3. 用户确认任务卡。
4. 系统生成大纲。
5. 用户确认大纲。
6. 用户点击“生成本节”。
7. 系统通过 HTTP RAG 获取材料并生成正文。
8. 前端通过 SSE/WebSocket 展示实时事件。
9. 用户选中段落并提出修改意见。
10. 系统生成 patch 预览，用户接受后提交版本。
```
